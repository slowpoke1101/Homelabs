Explanation/Blog: after learning about vpns and the cloud, during studying for the google it support certificate, i realized woah anyone could do this with basic stuff at home! so i did some thinking and research and landed on trying to run a wireguard vpn and nextcloud cloud services. i also learned about server operating systems and am interested in Linux so i decided to try to set everything up on an ubuntu server install. Also i found a cheap platform in the raspberry pi4 to be able to try this. at the time i only had a modest windows 11 laptop so i ordered a raspberry pi 4 and a samsung evo870 sata ssd to run the operating systom on, and in the meantime i had already began to tinker. i had harvested the hdd, fan, and lcd screen off of my old laptop from 2009, had a sata to usb enclosure so i did a practice run booting ubuntu server on the old hdd from my current laptop and installing nextcloud while i waited for the pi. 
	looking back i realize now that i could have downloaded virtualbox or VMware workstation to do this but i wasn't aware of virtualization yet, and my past experience with trying Linux was doing a dual boot back in the 2000's. Quick history, up until beginning of last winter i basically hadn't done anything with computers since back then, and the dual boot and online gaming were about as far as my tech knowledge went.
	Once i got the pi 4 the first reboot, after flashing my new ssd with ubuntu server 22 booting it from the pi, running update and upgrade, would fail. i tried different things and none of them worked and found out online that it was a known issue with certain sata to usb chip controllers not being able to regulate power draw from certain ssd's correctly. after ordering 3 different kinds of enclosures i finally got one that worked :))
	I was able to get wireguard running and client connections to my phone and laptop, also was able to configure nextcloud to only be available at the pi's wg0 ip address. it wasn't easy and i wish i had documented my steps from the beginning so i could go back and understand what wasn't clear to me back then. i was thrilled once these two services were up and running and wanted to try more! i found out about pihole adblocking and wanted to try that. this was very difficult and gave me a lot of trouble because i didn't understand what was going on yet. since i had already installed apache2 as the backend for nextcloud, i didn't understand that ports 80 and 443 were already taken and that i would need to reverse proxy the pihole's web ui in order to access it. so i settled for disabling the web ui and using padd, and dropping the blocklists into MariaDB manually since the pihole service was actually up and adblocking for my wireguard clients.
	I then found out about unbound and wanted to integrate that into the server so that unbound could serve dns to pihole.
once this stack was up and stable i really wanted to enable remote access since my plan after finding out about nextcloud was to remove all of my data from my then cloud provider and host it all on my own on nextcloud. i still didn't have a grasp on networking and basic hardening so i asked ai to check my system setup, provided it with what i wanted of the network, and then asked it help me harden everything so i could rest easy knowing that if i enabled a remote connection to my home lan, it was secure. i then learned basic hardening steps and ai gave me a list of firewall rules(ufw or uncomplicated firewall) to implement onto the pi4.
	What i ended up with was a patched together router with a faux subnet using wireguard, running nextcloud, pihole, and unbound all only accessible by the wireguard clients who were configured with a full tunnel. the rules i asked ai for were to prevent any traffic leaking from my wireguard tunnel to my home lan and vice versa unless explicitly allowed. 
	I continued to use and add to this setup after completing the google it support certificate and moved on to the CompTIA A+ exams.
i tuned the services, added fail2ban jails, rkhunter, logwatch and tried to think of a backup solution other than manual but dd required a bit for bit same size drive and rpi clone only works on rpi os's, rsync failed but i dont remember why so i settled for manual backups.



THE BEST PART about this server was learning iptables and packet flow through a Linux kernel!
Just like every new concept in IT I've had to tackle, when i found out about iptables and wanted to replace my ufw firewall, it felt like i'd never understand. i kept reading and watching videos for days and remained confused. why doesn't the packet go to the forward chain after the input chain? where does the packet come from before the output chain??? lol
It wasn't until i understood that a single host can have MULTIPLE network interfaces, and that each interface is basically like a 'separate profile' for the host. that mixed with learning that; 
a packet destined for the local machine goes to the input chain THEN to a service on the local machine, 
a packet passing through goes to the forward chain, and 
a packet originating from the local machine goes through output, they dont touch eachother, this UNLOCKED everything for me and reconfiguring the firewall with iptables has been by far one of the most fun things I've done and im so stoked i can read a firewall :) 
BUT no rest for the weary, i just learned that iptables is being depreciated so ill have to learn nftables with hook numbers idk what that means yet haha.
